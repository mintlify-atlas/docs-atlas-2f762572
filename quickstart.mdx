title: "Get started with GoReleaser Action"
description: "Learn how to add GoReleaser Action to your GitHub Actions workflow and automate Go project releases — from prerequisites to your first tag-triggered release."
---

GoReleaser Action integrates [GoReleaser](https://goreleaser.com/) into GitHub Actions with a single workflow step. This page walks you through creating a working release workflow — from prerequisites to pushing your first tag-triggered release.

<Steps>
  <Step title="Check prerequisites">
    Before adding the action, confirm you have the following in place:

    - A Go project with a valid `go.mod` file
    - A `.goreleaser.yaml` configuration file at the repository root (run `goreleaser init` locally to generate one)
    - A GitHub repository with Actions enabled

    If you haven't configured GoReleaser yet, see the [GoReleaser documentation](https://goreleaser.com/intro/) to generate a starting configuration.
  </Step>

  <Step title="Add the workflow file">
    Create `.github/workflows/release.yml` in your repository with the following content:

    ```yaml .github/workflows/release.yml
    name: goreleaser

    on:
      pull_request:
      push:

    permissions:
      contents: write

    jobs:
      goreleaser:
        runs-on: ubuntu-latest
        steps:
          -
            name: Checkout
            uses: actions/checkout@v6
            with:
              fetch-depth: 0
          -
            name: Set up Go
            uses: actions/setup-go@v6
          -
            name: Run GoReleaser
            uses: goreleaser/goreleaser-action@v7
            with:
              distribution: goreleaser
              version: '~> v2'
              args: release --clean
            env:
              GITHUB_TOKEN: ${{ secrets.GITHUB_TOKEN }}
    ```

    <Note>
      The `fetch-depth: 0` setting in the Checkout step is required. Without it, GoReleaser cannot access the full commit history and the changelog will be empty or incorrect.
    </Note>

    <Tip>
      The `version: '~> v2'` input controls which **GoReleaser binary** is installed. The `@v7` in `uses: goreleaser/goreleaser-action@v7` controls the **action** version. Keep both: pin the action to a major tag (`@v7`) and use `~> v2` for the GoReleaser binary to automatically receive patch updates within the v2 series.
    </Tip>
  </Step>

  <Step title="Verify the GITHUB_TOKEN secret">
    The `GITHUB_TOKEN` secret used in the workflow is provided automatically by GitHub Actions — you do not need to create or store it manually. GitHub generates a fresh token for each workflow run and scopes it to your repository.

    The `permissions: contents: write` block at the top of the workflow grants the token the write access GoReleaser needs to create and publish GitHub Releases. Without this, the release step will fail with a permission error.

    If you need to push release artifacts to a separate repository (for example, a Homebrew tap), you must use a [Personal Access Token](https://docs.github.com/en/authentication/keeping-your-account-and-data-secure/creating-a-personal-access-token) stored as a repository secret instead of `GITHUB_TOKEN`. See [Limitations](/advanced/limitations) for details.
  </Step>

  <Step title="Push a tag to trigger a release">
    With the workflow committed to your repository, push a version tag to trigger a full release:

    ```bash
    git tag v0.1.0
    git push origin v0.1.0
    ```

    GitHub Actions will start the `goreleaser` job automatically. GoReleaser will build your binaries, generate a changelog from commits since the previous tag, and publish a GitHub Release with the built artifacts.

    You can monitor the run under the **Actions** tab of your repository.
  </Step>
</Steps>
